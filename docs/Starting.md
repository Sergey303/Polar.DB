## Проект GetStarted и подготовка среды

Раздел начальных проб организован как набор простых программ в проекте `GetStarted`. Все программы
реализованы под `.NET Core`. Для их опробования надо подготовить рабочую среду `.NET Core`. Это
можно сделать либо через `Visual Studio`, либо через самостоятельную систему 
программирования с использованием SDK, и редактирования, например `Visual Studio Code`. Детали установки и основы работы можно посмотреть на сайте `asp.net`. Второй вариант можно рекомендовать, если вы легко работаете в командной строке и
вам не хочется устанавливать «монстра» `Visual Studio`.

Средствами `Visual Studio` или редактора создадим консольное приложение и проверим, что оно работает.

Библиотеку `Polar.DB` присоединим через `NuGet`-пакет `Polar.DB`, находящийся в стандартном
хранилище пакетов. Один из самых простых и надёжных способов — указать пакет в файле проекта
`.csproj`:

```xml
<ItemGroup>
    <PackageReference Include="Polar.DB" Version="2.1.1" />
    <!-- другие подключения -->
</ItemGroup>
```

Соответственно, полезно также подключить пространство имён `Polar.DB`.

<a id="starting-first-type"></a>
## Первые типы и структурные значения

Практически всегда, когда мы используем Поляр, нужно создать нужные поляровские типы. Это и будет первым шагом. Поляровский тип `PType` — это либо один из атомарных вариантов, либо запись, последовательность или объединение. Чтобы задать запись, состоящую из целого, строкового и
вещественного полей, можно использовать следующий фрагмент:

```csharp
PType tp = new PTypeRecord(
    new NamedType("id", new PType(PTypeEnumeration.integer)),
    new NamedType("name", new PType(PTypeEnumeration.sstring)),
    new NamedType("age", new PType(PTypeEnumeration.real)));
```

Если у нас есть поляровский тип, мы можем манипулировать структурными поляровскими значениями.
Наиболее «оперативная» форма представления таких значений — объектная. При этом поляровский
атомарный объект представляется объектом `C#` соответствующего типа: `integer -> int`, `sstring ->
string`, `real -> double`. А структуры представляются массивами поляровских объектов: `record ->
object[]`, `sequence -> object[]`, `union -> object[]`. Семантическая тонкость состоит в том, что
поляровские значения должны быть согласованы с поляровскими типами.

Пользуясь этими правилами, нетрудно сконструировать значение заданного типа `tp`, например:

```csharp
object value = new object[] { 998, "Пупкин Вася", 14.0 };
```

Что можно сделать со структурным значением? Можно выполнить ряд сериализаций: преобразовать
структурное значение в последовательность байтов, то есть сделать байтовую сериализацию, или в
последовательность текстовых символов, то есть сделать текстовую сериализацию. Есть и отладочная
текстовая сериализация, которая часто используется, чтобы «посмотреть» значение в динамике работы.
Так что тестовый фрагмент можно закончить выводом на консоль:

```csharp
Console.WriteLine(tp.Interpret(value));
```

Во фрагменте структурный объект интерпретируется в виде текста средствами соответствующего метода
поляровского типа. Более содержательное использование начинается при отображении структурных
объектов в поток байтов, то есть при записи объекта, и при извлечении структурных объектов из потока
байтов, то есть при чтении объекта. Обычно это делается через бинарную сериализацию и
десериализацию. Чтобы это стало понятнее, продолжим тест на примере последовательности записей
объединённого типа.

### Определение типа, структурные объекты, сериализации и десериализации

Существует синтаксическая форма для задания типов, напр.:
```text
//Figure = nothing ^ none,
//    point ^ Point,
//    polygon ^ [Point],
//    circle ^ {center: Point, radius: real};
//Point = {x: real, y: real};
```
Эта форма используется в иллюстративных целях, поскольку функциональные определения не всегда легко понятны. В определениях с имененм вводимого типа сопоставляется формула типа, в которой используются примитивные типы bool, byte, char, int, long, real, string. Также есть примитивный тип none, который соответствует множеству, не имеющему значений. Определение последовательности выполняется использованием квадратных скобок [формула_типа_элемента], определение записи выполняется перечислением полей записи с именами и типами, заключенным в фигурные скобки. Объединенный тип задается множеством пар тег ^ типовая_формула, разделителем, как и для записей является запятая.   

Смысл приведенного ранее определения в том, что фигура это или ничего (пустая) или точка или полигон, задаваемый последовательностьию точек или круг, задаваемый точкой центра и радиусом. Точка - естественно это запись из пары чисел. Этим определениям соответствуют определения в функциональной исполняемой форме:

```csharp
PType tp_point = new PTypeRecord(
    new NamedType("x", new PType(PTypeEnumeration.real)),
    new NamedType("y", new PType(PTypeEnumeration.real)));
PType tp_figure = new PTypeUnion(
    new NamedType("nothing", new PType(PTypeEnumeration.none)),
    new NamedType("point", tp_point),
    new NamedType("polygon", new PTypeSequence(tp_point)),
    new NamedType("circle", new PTypeRecord(
        new NamedType("center", tp_point),
        new NamedType("radius", new PType(PTypeEnumeration.real)))));
PType tp_sequ = new PTypeSequence(tp_figure);
```
В соответсвии с задаваемыми типами, можно создавать объекты в разном представлении. Текстовая форма представления похожа на JSON: примитивные значения изображаются традиционно: true/false для булевских, числа для байтов, символы в одинарных кавычках для символьных, целые и вещественные - как у всех, записи изображаются как наборы значений полей, снабженных именами полей, помещенными через запитую в фигурные скобки, последовательности - перечислением через запятую значений, помещенных в квадратные скобки. Значение объединенного типа изображается в виде идентификатора варианта и значением, соответствующему этому варианту. Кстати, значение типа none - это "пустышка", т.е. строка с нулевой длиной. 

Зададим в текстовом виде значение тиипа tp_sequ, т.е. изобразим последовательность фигур. К примеру, это будут точка, полигон и круг. 
```
//Sequ figs3 = 
//[
//    point^{ 3.5, 7.8 },
//    polygon^ [ point^{x: 0.0, y: 0.0 }, point^{x: 1.0, y: 0.0 }, point^{x: 1.0, y: 1.0 }, point^{x: 0.0, y: 1.0 } ],
//    circle^ { center: point^{x: 5.0, y: 5.0 }, radius: 4.99 }
//];
```
Следующее представление - объектное. В этом представлении структурное значение задается либо примитивным значеним языка реализации, в данном случае - C#, если оно само примитивно, либо массивом объектов. Для записи - это просто массив значений элементов, для последовательности - аналогично, это последовательности элеменитов, для объединени - это массив из двух элементов: первых номер варианта, второе - подзначение этого варианта. Объектное представление доступно для прямого ввода констант и значений. Например, предыдущий набор из трех фигур, можно задать: 

```csharp
object sequ = new object[]
{
    new object[] { 1, new object[] { 3.5, 7.8 } },
    new object[]
    {
        2,
        new object[]
        {
            new object[] { 0.0, 0.0 },
            new object[] { 1.0, 0.0 },
            new object[] { 1.0, 1.0 },
            new object[] { 0.0, 1.0 }
        }
    },
    new object[] { 3, new object[] { new object[] { 5.0, 5.0 }, 4.99 } }
};
```
Возможна другая форма текстового представления, ее можно назвать обратной польской записью. Это когда поток символов формируется по следующему правилу: примитивные значения превращаются в символы естественным образом типа: 44 -> "44", запись - это последовательность потоков элементов, последовательность, это сначала изображается (в виде строки) число элемнетов записи, а потом, перечисляются элементы. Значение объединенного типа разворачивается в номер варианта и значение варианта. Разделителем выступает, напр. пробел. Такая форма редко используется, к тому, же ее очень просто можно написать используя типовый объект и .ToString() к примитивным значениям. 

Еще одно представление структурных значений - размещение в поток байтов. Оно похоже на обратную польскую запись. В поток байтов помещаются следующим образом: примитивные пишутся в бинарном виде. В частности, булевской и байт пишутся в один байт, символ пишется (в кодировке Unicode) в 2 байта, целое - в 4 байта, элементы записи располагаются подряд без разделителя. Последовательность записывается в виде количества элементов и элементов. Количество элементов фиксируется длинным целым, сами элементы последовательности пишутся подряд (бинарно) и без разделителей. Значение объединенного типа записывается в виде байта с номером варианта и потом значения варианта. Для типа none, место не выделяется и не используется. 

Байтовое (бинарное) представление интересно своей копактностью и, для многих случаев, вычисляемостью координаты нужного элемента. Разные формы представления структурных значений служат разным задачам, но легко переходят из одной формф в другую. Промежуточной формой является объектное представление. Объектное представление может быть сериализовано в виде потока символов или потока байтов и эти потоки могут быть десериализованы, т.е. превращены в объектное представление. Естественно, такие действия обратимы - если объект сериализован, то после десериализации мы получим квивалентный объект. И наоборот, структура, зафиксированная в виде потока, после десериализации превратится в объект, который может быть сериализован с получением той же цепочки символов алфавита. 

Сделаем некоторые преобразования над последовательностью фигур и типом последовательности фигур:
```
// Выполним текстовую сериализацию
TextFlow.Serialize(Console.Out, sequ3, tp_sequ);
Console.WriteLine();

// Создадим Stream, сделаем бинарную сериализацию и будем считать, что это файл
Stream stream = new MemoryStream();
ByteFlow.Serialize(new BinaryWriter(stream), sequ3, tp_sequ);

// Десериализуем поток
stream.Position = 0L;
object sequ_1 = ByteFlow.Deserialize(new BinaryReader(stream), tp_sequ);

// Проверим полученное значение
Console.WriteLine(tp_sequ.Interpret(sequ_1));
```
Желательно предыдущие фрагменты кода собрать, исполнить и проанализировать результаты. 

Для чего все это нужно? Текстовое представление нужно для того, чтобы с помощью тектового редактора вводить и корректировать структурные значения. Также текст нужен в конце вычислений для анализа результатов. Бинарное представление хорошо подходит для хранения данных произвольного размера в файлах базы данных. Кроме того, в бинарном представлении есть понятие позиции (головки чтения) и методы установки позиции в заданное значение. Это значит, что в процессе записи можно определять позицию элемента, ее где-то фиксировать, а потом использовать для быстрого доступа к представлению этого элемента.  

Совсем просто использовать предложенные механизмы для конфигураторов и организации работы с ними. Определяем структурный объект, который сможет конфигурировать приложение. Структурный объект может содержать скалярные атрибуты, последовательности чего-то, назначение альтернатив с их альтернативными параметрами. Далее, задаем структурное значение в виде текстового файла, редактируемого текстовым редактором, в начале программы читаем этот файл, "раскидываем" параметры конфигурации по переменным.   

### Key-value хранилище
Сконструируем из библотечных элементов простое хранилище "ключ-значение". То есть, это будет набор однотипных объектов, помещаемых по уникальным ключам, взятых из значения.

В программе будет следующая последовательность действий: определение типа элемента, определение файловой базы, создание универсальной последовательности класса USequence с нужными настройками. Зафиксируем тип элемента как запись из трех полей: уникального целочисленного идентификатора, строкового имени, целочисленного поля "возраст":
```
// Тип элемента последовательности
PType tp_pers = new PTypeRecord(
    new NamedType("id", new PType(PTypeEnumeration.integer)),
    new NamedType("name", new PType(PTypeEnumeration.sstring)),
    new NamedType("age", new PType(PTypeEnumeration.integer)));

// Указываем директорию для файлов базы данных, формируем генератор потоков
string dbpath = "C:\\Home\\data\\getstarted\\";
int cnt = 0;
Func<Stream> GenStream = () => new System.IO.FileStream(dbpath + "f" + (cnt++) + ".bin", FileMode.OpenOrCreate, FileAccess.ReadWrite);

// Создаем универсальную последовательность
USequence useq = new USequence(tp_pers, dbpath + "state.bin", GenStream, ob => false,
    ob => (int)((object[])ob)[0], ic => (int)ic, false);
``` 
Посмотрим на используемые фактические параметры. Генератор потоков GenStream для универсальной последовательности позволяет создавать несколько файлов (потоков), требующихся в том или ином решении библиотеки. В данном случае, используется простое решение по формированию файлов по потребности. Еще один требуемый файл state.bin определяется напрямую через вводимое имя. Далее идет функция, определенная на объектном представлении элементов, означающая, что этой функцией элемент может быть "уничтожен", т.е. признан не действиельным. Полезное применение функции мы рассмотрим позже, а пока пусть функция ничего не делает. Далее, идут две функции, которые помогают эффективно индескировать по ключевому параметру. Первая - вычисляет ключевое значение, причем годится только сравнимые, т.е. обладающие интерфейсом IComparable. Обычно это или числовые или строковые. В данном случае, мы берем целочисленное значение поля id. Еще одна функция задает вычисление целочисленного хеш-значения на ключевом значении. В данном случае, это тождественное преобразование, но может быть и другим, для строковых ключей часто это хеш функция срока -> целое. Последний параметр включает, в данном варианте не включает, оптимизацию. 

Фактически, простое хранилище объектов типа tp_pers по ключу id уже создано. Только надо воспользоваться. Для этого, надо хранилище наполнить данными и можно будет запрашивать записи по значению ключа. Заполнение данными состоит из двух фаз: собственно загрузки и, второй, построения индекса. Простое тестовое решение для заполнения:
```
// Загрузка данными
int npersons  = 1000;
useq.Load(Enumerable.Range(0, npersons)
    .Select(i => new object[] { npersons - i - 1, i.ToString(), 33 }));
// Построение индекса
useq.Build();

// Испытание 
int key = npersons * 2 / 3;
var valu = useq.GetByKey(key);
// Проверка
Console.WriteLine(tp_pers.Interpret(valu));
```  
Комментарии: Ключи в последовательности идут в обратном порядке npersons - 1, npersons - 2, ...
Это для того, чтобы требовалась содержательная сортировка в формировании ключевой последовательности. Ключ для поверки выбирается из "серединки" и чтобы было по результату понятно, что "мы попали". Конструкция tip.Interperet(value) позволяет интерпредировать объектное представление value в контексте его типа. 

Собрав в единой программе два предложенных фрагмента, мы получаем простое key-value хранилище, заполняем его данными, вычисляем индекс, проверяем что хранилище выдает правильное значение. Программа без комментариев содержит 16 (!) строк. При этом можете сами проверить хранилище может хранить и обслуживать миллионы (на моем компьютере до 300 млн.) записей.  





<a id="starting-main2"></a>
## Main2()

Программа демонстрирует создание таблицы и простого сортировочного индекса для эффективного доступа
по идентификатору записи.

Теперь уже придётся «мусорить» в файловой системе: файлы, порождаемые в тестах, размещаются в
`./Databases`. Сами программы ничего не уничтожают, поэтому пользователю нужно самостоятельно
следить за файлами и вовремя их очищать. Не забудьте поменять вызываемый пример в `Main()` файла
`Program`.

В демонстрации создаётся таблица с типом последовательности записей из трёх полей, а также ещё один
тип — последовательность записей из двух полей. Затем формируются основная таблица и индекс. Сначала
создаётся пустая последовательность, затем в неё добавляются элементы в объектном структурном
представлении, а в конце обязательно вызывается `Flush()`, чтобы закончить построение массива на
диске.

После этого идёт проверка корректности работы поиска элемента по заданному целочисленному
идентификатору, а затем выполняется серия поисков с разными значениями идентификаторов. По авторским
измерениям получается около `54` мс на тысячу поисков.

Аналогичная задача выполняется в `MS SQL Server`, `MySQL` и `SQLite` заметно медленнее, хотя часть
проигрыша объясняется более тяжёлым запуском поиска в этих системах. Раздел формирования базы данных
и создания индекса выполняется в режиме `toload == true`. Один раз сформировав базу данных, можно
запускать программу далее в режиме `toload == false`; база данных очень быстро откликается на
запросы.

<a id="starting-main3"></a>
## Main3()

Программа демонстрирует решение предыдущей задачи, то есть «таблица + индекс», с использованием
специального универсального индекса `UniversalIndex`. Простой сортировочный индекс недостаточно
гибок и недостаточно эффективен. В универсальном решении заложены и гибкость, и эффективность.
Гибкость проявляется в том, что теперь позволено добавлять новые записи и после создания индекса.
Рост эффективности связан с добавлением так называемой шкалы — своего рода индекса к индексу.
Появляется третий файл.

Результаты прогона дают около 26 мс на тысячу поисков, что более чем в два раза лучше решения с
использованием простого индекса. Новую функциональность, связанную с добавлением и уничтожением
элементов в динамике доступа к ним, этот тест не демонстрирует.

<a id="starting-main4"></a>
## Main4()

Программа демонстрирует решение предыдущей задачи тем же подходом, то есть с использованием
универсального индекса, но уже в среде страничного хранилища `PagedFileStore` и с использованием
кеша этого хранилища.

Страничное хранилище используется несколько грубовато, но общая логика понятна. Все используемые
файлы теперь упакованы в одно хранилище — файл `storage.bin`. В реальных базах данных создаются
десятки таблиц, к ним — множество индексов, и вся эта картина ещё и меняется в динамике работы.

Производительность решения снова возрастает: теперь она достигает около 9 мс на тысячу поисков.
Правда, замедляется построение индексов. Чем-то приходится жертвовать. «Классические» `MS SQL
Server` и `MySQL` тоже не отличаются особенно быстрой загрузкой, хотя у них при этом есть поддержка
транзакций.

<a id="starting-main5"></a>
## Main5()

Программа демонстрирует некоторое модельное «предельное» решение предыдущей задачи. Как и раньше,
строится основная таблица и сортировочный индекс, но теперь индекс выполняется в виде двух
разнесённых по ячейкам последовательностей. Для использования из базы данных извлекается массив
ключей. Поиск происходит сначала в массиве ключей: там методом дихотомии ищется индекс элемента с
совпадающим ключом. Затем по этому индексу в третьей ячейке считывается единственный элемент — офсет
в основной последовательности. И последней операцией соответствующая запись читается по этому
офсету.

После построения базы данных выполняется тест производительности выборки по идентификатору.
Поскольку часы начинают измерять слишком малые промежутки времени, тест проводится не тысячу раз, а
сто тысяч раз. Результат снова улучшает скорость: `100 000` выборок выполняются примерно за `290`
мс, то есть около 2.9 мс на тысячу.

<a id="starting-main6"></a>
## Main6()

Программа продолжает выстраивание обработки последовательности записей `{id, name, age}`, но теперь
идентификатор определён как строковый. Показано, что решениями серии `UniversalIndex` можно
организовать обработку строкового поля `id` со скоростями, не слишком меньшими, чем в случае
целочисленного идентификатора. Время выполнения тысячи запросов на выборку записи по идентификатору
составляет около `30` мс.

Для данных размера `10` млн. записей рекомендуется провести дополнительные эксперименты. Базовый
прогон включает формирование базы данных и индексов, а затем выполнение в цикле запросов по случайно
сгенерированным идентификаторам. Далее можно изменить параметр `toload` на ложь и посмотреть
скорость доступа. Особенно интересен эксперимент с отключением загрузки кеша через строку
`ps_store.LoadCache()`, перезагрузкой компьютера и повторным выполнением серии доступов. Для `10`
млн. записей приводится результат около `88` секунд на `10 000` поисков, что резко контрастирует с
базовым прогоном, когда кеш активен из-за загрузки, вычислений или специального разогрева. Само
действие загрузки кеша для `10` млн. записей занимает около `3.6` секунды. Отсюда делается вывод:
разогрев в динамике работы не всегда эффективен, и в ряде случаев желательно проводить его
специально.
