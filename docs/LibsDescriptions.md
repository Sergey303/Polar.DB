# Описание библиотек Polar.DB

`Polar.DB` состоит из ряда библиотек и связанных с ними демонстрационных решений.

---

<a id="libs-overview"></a>
## Состав библиотек

- `Polar.DB` — базовая библиотека системы поляровской структуризации.
- `Polar.Cells` — ячейки для хранения структурированных данных, средства формирования ячеек и
  средства доступа к этим данным.
- `Polar.CellIndexes` — индексные построения для последовательностей и таблиц, вкладываемых в
  ячейки.
- `Polar.PagedStreams` — классы, предназначенные для формирования неограниченного множества
  потоков (`Stream`).
- `Polar.Datanode` — решения для распределённого хранения и обработки данных.
- `Polar.TripleStore` — готовое решение для хранилища триплетов.

Кроме библиотек, существует набор тестовых задач и простых решений, построенных на базе `Polar.DB`.
Пока этот набор сведён к проекту `GetStarted`, в котором каждая задача реализуется в виде своего
модуля `Main()`. В некоторых случаях к модулю добавляются дополнительные определения классов.

<a id="libs-polar-db"></a>
## Polar.DB: базовые средства

Поляровские типы присутствуют в программе в виде объектов класса `PType` и его производных
`PTypeRecord`, `PTypeSequence` и `PTypeUnion`. Поляровские структурные объекты являются древовидными
построениями и интерпретируются только в контексте типа. Соответственно, при обработке явно или
неявно собирается пара `{ v, t }`, где `v` — структурное значение, `t` — тип этого значения. Сам тип
также представляет собой древовидное построение, к которому применима поляровская структуризация.
Соответственно, существует константный объект «тип типов» `TType`.

Типовая система Поляра содержит некоторое отладочное средство для визуализации структур. Текстовый
вариант значения получается применением к значению класса `PType` метода `Interpret(object v)`, где
`v` — объектное представление структурированного значения.

Следующая группа действий, определяемая в `Polar.DB`, — сериализации. Сериализация — это
представление структурного значения в виде последовательности некоторых «букв». Десериализация —
обратный процесс, то есть получение объекта из цепочки букв. Есть две пары
сериализация/десериализация: текстовая и бинарная. При текстовой сериализации «буквами» являются
символы Unicode, при бинарной — байты. Модули `TextSerialize` и `BinarySerialize` дают процедуры
прямого и обратного преобразований. Существенной чертой этих процессов является рекуррентность:
сериализация структурного значения выполняется через сериализацию его элементов с добавлением
связующих элементов.

При этом у каждой серии «букв» есть конкретный адрес; для байтов это смещение (`offset`) в потоке.
Таким образом, если зафиксировать начало серии и начинать чтение с этой точки, мы будем получать
копию записанного там структурного объекта.

### Пример работы со `Stream`

```csharp
PType tp = ...;
object v1 = new object[] { ... };
Stream stream = File.Open(...);
BinaryWriter bw = new BinaryWriter(stream);

// Где-то сохраняем позицию записи
long offset1 = stream.Position;

// Записываем значение в поток
Serialize(bw, v1, tp);

// Теперь, когда значение понадобилось, установим позицию чтения и прочитаем
stream.Position = offset1;
BinaryReader br = new BinaryReader(stream);
object v2 = Deserialize(br, tp);

// Значение v2 будет таким же, как v1.
```

Эта схема показывает, как отдельный `Stream` или множество потоков можно использовать как память, в
которую можно положить структурное значение, а потом из этого места неограниченное число раз его
извлекать. Схема выглядит громоздкой, но она весьма эффективна и универсальна; на практике она
используется через классы и методы более высокого уровня.

<a id="libs-universal-sequence"></a>
## UniversalSequenceBase

Одним из часто используемых частных случаев является последовательность элементов определённого
типа. Последовательность элементов возникает как задача накопления, хранения и обработки данных, а
также при построении дополнительных структур — индексов, которые делают доступ к данным эффективным.
Последовательность элементов одного типа можно задать классом `UniversalSequenceBase`, параметрами
конструктора которого являются тип элемента и `Stream` — среда бинарной реализации.

```csharp
PType tp_el = new PTypeRecord(
    new NamedType("id", new PType(PTypeEnumeration.integer)),
    new NamedType("name", new PType(PTypeEnumeration.sstring)),
    new NamedType("age", new PType(PTypeEnumeration.real)));

Stream media = File.Open(...);
var ubs = new UniversalSequenceBase(tp_el, media);
```

Такую последовательность можно очищать, заполнять данными, добавлять отдельные элементы, читать
элементы с установленной позиции, а в определённых ситуациях — записывать элементы или их части.
Последовательность можно сканировать, то есть последовательно читать элементы с начала и до конца.

Существенным моментом является фиксированность или нефиксированность размера элемента. Структурное
значение обладает фиксированным размером, если фиксированным размером обладают все его элементы;
почти все простые типы обладают фиксированным размером, кроме строки. Если элемент фиксированного
размера, то можно легко вычислить офсет `i`-го элемента по формуле:

```text
offset = start + i * elementsize
```

Для элементов нефиксированного размера позицию в потоке также можно определить, но только через
сканирование, что для большинства случаев неприемлемо медленно.

Ещё одно действие, задаваемое только для последовательностей фиксированного размера, — сортировка.
Например, метод `Sort32(Func<object, int> keyFun)` выполняет сортировку последовательности
относительно ключевой функции, заданной в аргументе. Для 64-разрядного случая предусмотрен
аналогичный метод `Sort64`.

<a id="libs-indexes"></a>
## Индексы

Наличие сортировки позволяет преодолеть трудности прямого доступа для последовательностей с
нефиксированным размером элементов. Делается это по следующей схеме: кроме основной, то есть
опорной, последовательности, параллельно создаётся последовательность пар `{ key, offset }`,
содержащая значения ключей и офсеты элементов, размещённых в опорной последовательности. Эта
индексная последовательность имеет фиксированный размер элементов и может быть отсортирована по
значению ключа. После этого методом деления пополам, то есть дихотомии, нужный для заданного ключа
элемент получается за логарифмическое относительно размеров последовательности время.

Реализация такого подхода имеет ряд алгоритмических и технологических тонкостей, поэтому сведена к
основным случаям в библиотеке `Polar.DB`: `IndexKey32Comp`, `IndexView`, `IndexKey32CompVector`.

### `IndexKey32Comp`

При определении универсального ключевого индекса `IndexKey32Comp` задаются генератор потоков,
носитель данных индекса, опорная последовательность, предикат применимости индекса к данному
элементу, хеш-функция, заданная на элементах опорной последовательности, и компаратор, сравнивающий
элементы опорной последовательности.

Индекс работает следующим образом: элементы последовательности упорядочиваются сначала в порядке,
задаваемом хеш-функцией, а вторичный порядок задаётся компаратором. Хеш-функция не всегда является
именно «хешем», а компаратор может быть равен `null`, что означает его отсутствие. Индекс может быть
построен (`Build`), после чего к нему можно обращаться с запросом обо всех элементах,
соответствующих некоторому образцу.

### `IndexView`

Индекс `IndexView` создаётся указанием среды хранения, опорной последовательности, функции
применимости и компаратора. В результате индекс определяет элементы, которые при сравнении с
образцом дают `0`.

### `IndexKey32CompVector`

Векторный индекс `IndexKey32CompVector` отличается тем, что каждый элемент порождает не один
индексный элемент, а несколько. Поэтому ему не нужна отдельная функция `applicable`: для
неприменимых элементов достаточно сформировать пустой результат. В остальном он близок к
`IndexKey32Comp`.
