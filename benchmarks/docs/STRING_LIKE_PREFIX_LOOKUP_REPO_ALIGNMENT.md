# Repo alignment notes for string LIKE lookup

Этот файл фиксирует, какие изменения текущего состояния PolarDB учтены в эксперименте.

## 1. Load / Build / Reopen разделены

Старый вариант архива измерял подготовку как одну фазу. Теперь это рискованно: в текущем репозитории порядок `Build()` важен сам по себе.

Эксперимент должен измерять отдельно:

1. `Load` — запись данных;
2. `Build` — стабилизация последовательности и построение индекса;
3. `Reopen` — закрытие и повторное открытие storage;
4. `Lookup` — запросы после reopen.

## 2. PolarDB Build должен уважать новый порядок

Реализация `IPolarStringLikeStore.BuildAsync` должна использовать текущий безопасный порядок:

1. зафиксировать основную последовательность;
2. построить индексы;
3. зафиксировать индексы;
4. сохранить корректное состояние.

Не надо строить индекс по полузавершённому состоянию данных.

## 3. Reopen проверяет logical end и state

`ReopenAsync` нужен не только как performance-фаза. Он защищает эксперимент от случая, когда lookup быстрый только в уже открытом процессе, но после рестарта storage вынужден дорого или неверно восстанавливать состояние.

Для PolarDB это особенно важно из-за текущих инвариантов:

- `AppendOffset` — logical end последовательности;
- state хранит count и logical end;
- refresh/recovery не должны слепо доверять `Stream.Length`;
- garbage tail не считается валидными данными.

## 4. Prefix lookup должен стартовать с first-match boundary

Для prefix-сценариев PolarDB должен искать первую позицию диапазона, а не произвольный duplicate внутри равного блока.

Правильная схема:

```text
lower = prefix
upper = next ordinal string after prefix
start = first key >= lower
while key < upper and key starts with prefix: count++
```

Именно это делает сравнение честным относительно SQLite `LIKE 'prefix%'` с индексом.

## 5. Contains — отдельный scan-сценарий

`LIKE '%text%'` не должен сравниваться с индексным prefix lookup как будто это та же операция.

В эксперименте `containsScan` оставлен, но его надо анализировать отдельно.
