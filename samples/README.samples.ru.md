# Samples

Эта папка содержит публичные учебные примеры Polar.DB.

Цель этих проектов — не сохранить каждый исторический эксперимент, а дать понятный маршрут изучения актуальной публичной поверхности репозитория.

## Рекомендуемый порядок изучения

1. **GetStarted.StructuresAndSerialization**
   Начинать стоит отсюда. Этот проект знакомит с типами Polar, объектным представлением значений, интерпретацией и текстовой/бинарной сериализацией.

2. **GetStarted.SequencesAndStorage**
   Следующий шаг после первого проекта. Здесь показывается, как записи хранятся в последовательностях, как работают append/flush/scan и как в модель хранения вписывается доступ по offset.

3. **GetStarted.IndexesAndSearch**
   Завершающий учебный проект. Он показывает актуальную поверхность индексов и поиска: первичный поиск по ключу, вторичные индексы, текстовый поиск, поиск по множественным значениям, а также вспомогательные утилиты вроде `Scale` и `Diapason`.

## Принципы этих примеров

- Каждый проект тематический.
- Сценарии достаточно небольшие, чтобы их можно было прочитать за один заход.
- Runtime-данные создаются локально рядом с результатом сборки sample-проекта.
- Примеры должны объяснять библиотеку, а не скрывать её за лишней обвязкой.

## Чего здесь сознательно нет

Эта папка — публичная учебная поверхность. Здесь не место legacy-оболочкам, миграционным остаткам и большим разовым экспериментам.

Если старый код нужен для истории или сравнения, его лучше держать вне основного учебного пути.

## Типовые команды

```bash
dotnet run --project samples/GetStarted.StructuresAndSerialization -- list
dotnet run --project samples/GetStarted.SequencesAndStorage -- list
dotnet run --project samples/GetStarted.IndexesAndSearch -- list
```

Большинство sample-проектов придерживаются одного и того же стиля команд:

```bash
dotnet run -- list
dotnet run -- all
dotnet run -- <scenario-id>
```

## Заметка для контрибьюторов

Публичные samples должны оставаться простыми, актуальными и тематически чистыми.

Когда добавляется новый сценарий:
- лучше одна ясная идея на один сценарий;
- комментарии должны помогать читателю, который видит код впервые;
- не стоит дублировать другой сценарий только с косметическими отличиями;
- лучше использовать актуальные API и актуальные пространства имён.
