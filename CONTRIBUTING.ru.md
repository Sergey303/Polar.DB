# Вклад в Polar.DB

Спасибо за интерес к Polar.DB.

Этот репозиторий содержит библиотеку .NET для типизированных структурированных данных, append-ориентированного хранения последовательностей и индексного доступа. Вклад приветствуется, но корректность для этого проекта важнее поверхностных изменений ради изменения.

## Перед началом

Пожалуйста, сначала прочитайте:

- [README.md](README.md)
- [docs/REPOSITORY_STATE.md](docs/REPOSITORY_STATE.md)
- [SECURITY.md](SECURITY.md)

Если изменение затрагивает семантику хранения, recovery, индексацию или публичное API, сначала сверяйтесь с текущими инвариантами репозитория.

## Окружение разработки

Версия SDK фиксируется через `global.json`.

Рекомендуемое окружение:

- .NET SDK, совместимый с `global.json`
- Git
- Visual Studio 2022 / Rider / VS Code или другой редактор с хорошей поддержкой .NET

Сборка и тесты:

```bash
dotnet restore
dotnet build Polar.DB.sln
dotnet test tests/Polar.DB.Tests/Polar.DB.Tests.csproj
```

## Структура репозитория

- `src/Polar.DB/` — исходный код библиотеки
- `tests/Polar.DB.Tests/` — регрессионные и поведенческие тесты
- `samples/` — примеры
- `docs/` — документация по проекту и репозиторию

## Что мы принимаем

Приветствуются:

- исправления багов;
- тесты на исправленное поведение;
- улучшения документации;
- улучшения sample-проектов;
- упрощение API без ломки задокументированного поведения;
- аккуратно обоснованные улучшения производительности или модели хранения.

## Каким должен быть хороший вклад

Хорошее изменение:

- решает одну логическую задачу;
- покрыто тестами;
- сопровождается обновлением документации, если меняется публичное поведение;
- явно описывает риски и совместимость;
- уважает исторические ограничения дизайна и текущие инварианты.

## Ожидания к pull request

Каждый pull request должен:

- объяснять **что изменено**;
- объяснять **зачем это изменено**;
- объяснять **как это проверялось**;
- явно указывать, изменение это breaking, non-breaking или просто уточнение поведения;
- обновлять документацию, если меняется публичное поведение, поддерживаемый сценарий или ограничения.

Для исправлений багов по возможности добавляйте регрессионный тест.

## Правила по модели хранения и корректности

Для этого репозитория особенно важны такие правила:

1. **Не путайте физическую позицию потока и логическую границу валидных данных.**
2. **Не считайте, что `Stream.Length` всегда равен корректному логическому концу данных.**
3. **Считайте `AppendOffset` логической границей append, а не просто изменяемым курсором.**
4. **Не предполагайте, что любой variable-size in-place overwrite безопасен.**
5. **Данные должны быть стабилизированы до финализации индексов и состояния.**
6. **Где это повышает ясность, предпочитайте семантический доступ к полям вместо магических индексов.**

Любые изменения в recovery, refresh, append-логике, rewrite-поведении и индексах нужно делать консервативно и сопровождать тестами.

## Публичный API и стиль документации

Для новых публичных материалов:

- используйте каноническое имя `Polar.DB`;
- исторические ссылки сохраняйте только там, где они действительно нужны;
- community-health файлы верхнего уровня лучше вести на английском;
- историческую документацию не ломайте без осознанного плана миграции;
- не завышайте зрелость проекта и обещаемые гарантии.

## Стиль коммитов

Жёсткий формат коммитов не обязателен, но понятные сообщения очень желательны.

Хорошие примеры:

- `Fix first-match boundary handling in UKeyIndex`
- `Add regression tests for recovery with garbage tail`
- `Clarify AppendOffset semantics in README`

## Ломающие изменения

Breaking changes допустимы только при явном обосновании.

Такой PR должен:

- явно помечать себя как breaking;
- описывать миграционное влияние;
- обновлять документацию;
- не смешивать в одном PR unrelated-рефакторинг.
